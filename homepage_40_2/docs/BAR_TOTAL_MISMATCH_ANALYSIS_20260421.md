# 🔬 바(Bar) 총합 vs 준비+평가 합 불일치 정밀 분석 보고서

**버전**: v39.2.1 (프로덕션 실측)
**분석일자**: 2026-04-21
**분석 범위**: CCRA권장일수·KOIST프로세스 **바 위에 표시되는 총 개월수** vs **바 안의 준비+평가 개월수 합**
**분석 방법**: Playwright로 프로덕션 사이트(`https://koist-website.pages.dev`) 직접 자동화 측정 (4개 EAL × 9개 슬라이더 포지션 = 36 × 2(CCRA/KOIST) = 72 데이터 포인트)
**수정 조치**: ❌ 본 보고서에서는 **코딩 변경 없음** (요구사항: 분석만 수행)

## 🎯 최종 결론 (한 줄 요약)

> **CCRA 36개 포인트 중 13개(36.1%), KOIST 36개 포인트 중 5개(13.9%)** 에서 **"바 위 총합 ≠ 준비+평가 합"** 의 시각적 모순이 실제 프로덕션에서 발생 중이며, 추가로 **Saving 9개(25%)** 도 바 값과 어긋남.
>
> **원인**: 실수 계산값 `gPrepF`, `gEvalF`, `gTotalF` 를 각각 **독립적으로 `Math.round()`** 하여 화면에 뿌리기 때문. 수학적으로 `round(a) + round(b) ≠ round(a+b)` 이 성립하는 경우가 존재함.
>
> **해결 가능성**: **3개 옵션(B/C/D) 모두 72개 포인트 전수 100% 정합 보장**. 단 **1줄 수정**으로 해결 가능.

## 1. 사용자가 관찰한 현상 (문제 정의)

홈페이지 AI 시뮬레이터 영역에는 다음 **3개의 바(bar)** 가 있습니다.

```
┌─────────────────────────────────────────────────────┐
│  CCRA 권장일수              ⟨약 X개월⟩ ← 바 위 총합 │  ← 여기 숫자와
│  ┌──────────────────────────────────────────────┐  │
│  │ 주황(준비 N개월)  │ 회색(평가 M개월)          │  │  ← 여기 (준비+평가)의 합이
│  └──────────────────────────────────────────────┘  │     같아야 하는데
└─────────────────────────────────────────────────────┘     **다릅니다**.

┌─────────────────────────────────────────────────────┐
│  KOIST 평가프로세스          ⟨약 Y개월⟩              │
│  ┌──────────────────────────────────────────────┐  │
│  │ 주황(준비 P개월)  │ 파랑(평가 Q개월)          │  │
│  └──────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│  (하단) "약 S개월 절감 · 원스톱 서비스"              │  ← 이 값도
└─────────────────────────────────────────────────────┘     gT-kT와 안 맞음
```

**요구사항**: 사용자가 바 위에 표시된 "약 X개월"을 볼 때, 바 안의 "준비 N개월 + 평가 M개월" 을 시각적으로 더해서 같은 값이 나와야 함. 산술 정합성 확보.

```javascript
// 참고: d.saving = gTotalF - kTotalF (라인 1897에서 정의)
//                 → 실수 뺄셈 후 반올림
```

여기서 `fmtM()` 은 라인 1901에 정의:
```javascript
function fmtM(v) { return Math.round(v).toString(); }
```
→ 실질적으로 **`Math.round()`와 동일**.

### 3-2. 수학적 원인 — Rounding Sum Inequality

실수 a, b에 대해:
```
round(a) + round(b) ≠ round(a + b)  (최대 1 오차 발생 가능)
```

**구체적 예시 (EAL2, Prep=1%)**:
```
내부 실수값:  gPrep = 4.60,   gEval = 3.70,   gTotal = 8.30
독립 반올림:  round(4.60)=5,  round(3.70)=4,  round(8.30)=8

→ 화면 표시:  "준비 5개월" | "평가 4개월" | 바 위 "약 8개월"
→ 사용자 시각: 5 + 4 = 9 (그런데 총은 8)  ⚠ **1개월 괴리**
```

**3-Way Inconsistency (EAL3, Prep=60%)**:
```
내부 실수값:  gPrep=4.55, gEval=3.75, gTotal=8.30
독립 반올림:  round(4.55)=5, round(3.75)=4, round(8.30)=8

화면:  "준비 5개월 + 평가 4개월" = 9   vs   바 위 "약 8개월"
```

### 3-3. Saving 불일치 추가 원인

```javascript
// saving = gTotalF - kTotalF  (실수 뺄셈)
// 화면 표시 = Math.round(saving)

// 그러나 사용자는 "바 위 gTotal - 바 위 kTotal" 으로 직관적으로 계산
```

```

→ Saving 표시 6 ≠ 바 뺄셈 7  ⚠
```

## 5. 🎯 권장안 — **Option C** 가 최적

### 5-1. 권장 이유

1. **수정 최소** — `simulate()` 함수 내부 1줄만 바꾸면 끝
2. **디자인 무변경** — 기존 "약 N개월" 표시 그대로
3. **덧셈 100% 보장** — 72개 포인트 전 구간 정합
4. **시각적 직관성** — 사용자가 바를 눈으로 볼 때 "준비+평가=총합" 이 항상 맞음
5. **Total 오차 최대 1개월** — 실제 계산값과 1개월 이내 차이인데, 이미 현재도 동일 수준 오차 있음

### 5-2. (참고) 수정 지점 — 코딩은 다음 지시 시 진행

**파일**: `src/templates/home.tsx`, 라인 1889~1898 `simulate()` 반환부
```javascript
return {
  general: { prep: gPrepF, eval: gEvalF, total: gTotalF },
  ...
}
```

→ 다음과 같이 변경 (단, 현재는 분석만. 코드 변경은 지시 후 진행):

```javascript
// [변경 후 상상도 — 실제 반영 X]
var rgP = Math.round(gPrepF);
var rgE = Math.round(gEvalF);
var rkP = Math.round(kPrepF);
var rkE = Math.round(kEvalF);
return {
  general: { prep: gPrepF, eval: gEvalF, total: gTotalF,
             displayPrep: rgP, displayEval: rgE, displayTotal: rgP + rgE },
  koist:   { prep: kPrepF, eval: kEvalF, total: kTotalF,
             displayPrep: rkP, displayEval: rkE, displayTotal: rkP + rkE },
  ...
  displaySaving: (rgP + rgE) - (rkP + rkE),  // Saving도 자동 정합
}
```

그리고 `updateChart()`의 6줄(1927,1929,1931,1942,1944,1946,1955)을 `d.general.displayTotal` 등으로 교체.

### 5-3. 수정 적용 시 결과 예측 (EAL2, Prep=1%)

|---|---|---|

| CCRA 준비+평가 | 5+4=9 | 5+4=9 ✓ 일치 |
| KOIST 바 위 | 약 4개월 | 약 4개월 |
| KOIST 준비+평가 | 2+2=4 | 2+2=4 ✓ 일치 |
