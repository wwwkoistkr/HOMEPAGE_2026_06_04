### 2.2 불일치의 수학적 원인

`Math.round(a) + Math.round(b) ≠ Math.round(a + b)` 현상

**수학적 공리**:
> 두 실수 a, b의 소수부 합이 0.5를 넘거나 0.5에 걸치면,
> 각자 반올림한 합과 합을 반올림한 값 사이에 최대 ±1의 차이가 발생한다.

**EAL2, 슬라이더 1% 구체 예시**:
| 항목 | 실수값 (계산값) | 독립 반올림 | 결과 |
|---|---:|---:|---|
| 준비 (gPrep) | 4.595 | **5** | |
| 평가 (gEval) | 3.690 | **4** | |
| 합 (gTotal) | 8.285 | **8** ← v39.2 | ❌ **불일치** (5+4=9 ≠ 8) |
| **Option C 합** | 8.285 | **9** ← v39.3 | ✅ **일치** (5+4=9 = 9) |

### 2.3 EAL2·EAL4가 특히 두드러진 이유

#### ▶ EAL2 최대값 = 36주 ÷ 4.345 ≈ **8.285개월**
```
gPrep = 8.285 × 0.554 = 4.590  → round = 5
gEval = 8.285 × 0.446 = 3.695  → round = 4
              소수부 합 = 0.590 + 0.695 = 1.285 (>0.5)
              ⇒ round(prep) + round(eval) = 9
              ⇒ round(total)              = 8   ❌
```
**EAL2는 최대값이 소수점 0.285로 딱 떨어져서 경계값에 자주 놓임**

#### ▶ EAL4 최대값 = 64주 ÷ 4.345 ≈ **14.730개월**
```
gPrep = 14.730 × 0.554 = 8.160  → round = 8
gEval = 14.730 × 0.446 = 6.570  → round = 7
              소수부 합 = 0.160 + 0.570 = 0.730
              ⇒ round(prep) + round(eval) = 15
              ⇒ round(total)              = 15  ✓ (우연히 일치)

하지만 중간 슬라이더 50% 지점:
gPrep = 6.390, gEval = 5.150, gTotal = 11.540
              round = 6, 5, 12
              ⇒ 6+5=11 ≠ 12  ❌
```
**EAL4는 값의 폭이 커서(최대 64주 → 14.7개월) 슬라이더 이동 시 0.5 경계 교차 빈도가 높음**

#### ▶ EAL3가 상대적으로 덜 눈에 띄는 이유
EAL3는 48주(≈11.048)로 소수부가 0.048로 작고, 28-48 주 범위에서 중간값들이 정수 근처에 안정적으로 위치하여 불일치 발생이 EAL2/EAL4보다 적음.

## 5. 적용한 수정 방안: Option C

### 5.1 수정 전후 코드 비교 (`src/templates/home.tsx`)

#### ❌ v39.2 이전 (문제)
```javascript
// simulate() return
return {
  general: { prep: gPrepF, eval: gEvalF, total: gTotalF },
  koist:   { prep: kPrepF, eval: kEvalF, total: kTotalF },
  ...
};

// updateChart()
gTotal.textContent = '약 ' + Math.round(d.general.total) + '개월';  // round(a+b)
gPrep.textContent  = '준비 ' + Math.round(d.general.prep) + '개월'; // round(a)
gEval.textContent  = '평가 ' + Math.round(d.general.eval) + '개월'; // round(b)
// ⇒ round(a)+round(b) ≠ round(a+b) 현상 발생
```

#### ✅ v39.3 수정 (Option C 적용)
```javascript
// simulate() 내부에 display 전용 필드 추가
var rgP = Math.round(gPrepF);
var rgE = Math.round(gEvalF);
var rkP = Math.round(kPrepF);
var rkE = Math.round(kEvalF);
var gDisplayTotal = rgP + rgE;   // ← 핵심: "합이 먼저, 반올림은 부분만"
var kDisplayTotal = rkP + rkE;
var displaySaving = gDisplayTotal - kDisplayTotal;

return {
  general: {
    prep: gPrepF, eval: gEvalF, total: gTotalF,
    displayPrep: rgP, displayEval: rgE, displayTotal: gDisplayTotal  // ← 신규
  },
  koist: {
    prep: kPrepF, eval: kEvalF, total: kTotalF,
    displayPrep: rkP, displayEval: rkE, displayTotal: kDisplayTotal  // ← 신규
  },
  ...
  saving: gTotalF - kTotalF,
  displaySaving: displaySaving                                       // ← 신규
};

// updateChart() — display* 값만 사용
gTotal.textContent = '약 ' + d.general.displayTotal + '개월';        // ← 수정
gPrep.textContent  = '준비 ' + d.general.displayPrep + '개월';        // ← 수정
gEval.textContent  = '평가 ' + d.general.displayEval + '개월';        // ← 수정
```

### 5.2 Option C의 설계 원칙

1. **바 너비 계산은 실수값 유지** (`gTotalF`, `kTotalF`)
   → 시각적 연속성 유지 (바가 미세하게 움직임)
2. **단축률(reduction%)도 실수값 기반**
   → 정확한 비율 유지 (변경 없음)
3. **표시값은 반드시 `round(부분)의 합`**
   → 사용자가 보는 숫자의 산술 정합성 100% 보장
4. **내부 실수값은 보존** (총합, 절감 계산 등 내부 로직용)
