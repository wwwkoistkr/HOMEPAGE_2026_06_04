# KOIST 평가현황 시스템 설계서 v2.0

## 문서 정보
- **버전**: 2.0
- **작성일**: 2026-04-10
- **목적**: 평가현황을 10개 사업분류별로 완전 분리 관리하는 시스템 재설계

---

## 1. 현황 분석 및 문제점

### 1.1 현재 문제
| 문제 | 설명 |
|------|------|
| 사업 분류 미분리 | 모든 평가현황이 CC평가 하나로만 표시됨 |
| category 필드 미활용 | DB에 category 컬럼이 있으나 모두 'CC평가'로 동일 |
| 프론트 필터 없음 | 사업별 탭/필터가 없어 사업 구분 불가 |
| 관리자 분류 제한 | 관리자 폼에서 4가지 사업만 선택 가능 (CC평가, 보안기능시험, KCMVP, 성능평가) |
| 사업별 테이블 구조 차이 무시 | CC평가는 보증등급/인증구분이 있지만, 다른 사업은 다른 필드가 필요 |

### 1.2 실제 KOIST 사업 분류 (10개)
| # | 사업명 | 코드(slug) | 특성 |
|---|--------|-----------|------|
| 1 | CC평가 | cc | 보증등급(EAL), 인증구분, 신청구분 |
| 2 | 보안기능확인서 | security-func | 확인서 종류, 발급번호 |
| 3 | 암호모듈검증(KCMVP) | kcmvp | 암호모듈명, 검증번호, 알고리즘 |
| 4 | 성능평가 | performance | 성능등급, 평가항목 |
| 5 | 보안적합성검증 | compliance | 검증대상, 검증기준 |
| 6 | 취약점분석평가 | vulnerability | 분석유형, 위험등급 |
| 7 | 정보보호제품평가 | product-eval | 제품유형, 평가기준 |
| 8 | 클라우드보안인증 | cloud-security | 인증등급, 서비스유형 |
| 9 | IoT보안인증 | iot-security | 인증등급, 기기유형 |
| 10 | 기타시험평가 | etc | 일반 필드 |

## 2. 시스템 설계

### 2.1 DB 스키마 (변경 불필요)
기존 `progress_items` 테이블 그대로 활용. category 필드로 10개 사업 구분.

```sql
-- 기존 테이블 구조 (변경 없음)
progress_items (
  id, category, product_name, company, status,
  assurance_level, cert_type, eval_type,
  start_date, end_date, note, sort_order, created_at
)
```

- `category`: 사업 분류명 (예: 'CC평가', '보안기능확인서', 'KCMVP' 등)
- `assurance_level`: CC에서는 EAL등급, 다른 사업에서는 등급/번호 등으로 유연 활용
- `cert_type`: CC에서는 인증구분, 다른 사업에서는 유형/종류로 활용
- `eval_type`: CC에서는 신청구분, 다른 사업에서는 기준/방식으로 활용
- `note`: 추가 정보 (모든 사업 공통)

### 2.2 사업별 필드 매핑
| 사업 | assurance_level 용도 | cert_type 용도 | eval_type 용도 |
|------|---------------------|---------------|---------------|
| CC평가 | 보증등급(EAL1~7) | 최초평가/재평가 | 국내/국제평가 |
| 보안기능확인서 | 확인서 등급 | 발급구분 | 시험유형 |
| KCMVP | 검증등급 | 모듈유형 | 알고리즘 |
| 성능평가 | 성능등급 | 평가구분 | 평가항목 |
| 보안적합성검증 | 적합등급 | 검증구분 | 검증기준 |
| 취약점분석평가 | 위험등급 | 분석유형 | 평가범위 |
| 정보보호제품평가 | 평가등급 | 제품유형 | 평가기준 |
| 클라우드보안인증 | 인증등급 | 서비스유형 | 인증기준 |
| IoT보안인증 | 인증등급 | 기기유형 | 인증기준 |
| 기타시험평가 | 등급 | 유형 | 기준 |

## 3. 프론트엔드 설계

### 3.1 평가현황 페이지 (`/support/progress`)
- **상단**: 사업별 탭 네비게이션 (전체 + 10개 사업)
- **검색바**: 카테고리 필터 + 상태 필터 + 제품명 검색
- **테이블**: 카테고리에 따라 컬럼 헤더 라벨 동적 변경
- **페이지네이션**: 기존 구조 유지

### 3.2 홈페이지 미니 테이블
- 카테고리 탭 추가 (최근 5건씩 표시)
- 기본 탭: '전체' (최근 5건)

### 3.3 서비스 서브페이지 (`/services/:slug/progress`)
- 해당 사업부서의 카테고리만 자동 필터링

---

## 4. 관리자 설계

### 4.1 평가현황 관리 (`/admin/progress`)
- **카테고리 탭**: 사업별로 데이터 필터링
- **추가/수정 폼**: 선택된 카테고리에 따라 필드 라벨/옵션 동적 변경
- **일괄 작업**: 카테고리별 현황 통계 표시

---

## 5. 구현 범위

### Phase 1: 핵심 변경
1. 카테고리 상수 정의 (공통 모듈)
2. 프론트 평가현황 페이지에 카테고리 탭/필터 추가
3. 관리자 폼에 10개 카테고리 및 동적 필드 라벨
4. 홈페이지 미니 테이블에 카테고리 탭
5. 시드 데이터: 10개 사업별 샘플

### Phase 2: 라우트 변경
1. `/support/progress?category=CC평가` 쿼리 파라미터 추가
2. 서비스 서브페이지 자동 카테고리 매핑
3. API에 카테고리 필터 추가
