## 4. 단위 변환 (cm → px) ⭐ 핵심 분석

### 4.1 CSS px와 실제 cm의 관계

CSS에서 `1cm ≈ 37.8px` (96 DPI 기준)이지만, **이건 인쇄용 정의**입니다.
화면에서는 **모니터 DPI / 브라우저 줌 / 사용자 디스플레이 설정**에 따라 다릅니다.

| 모니터 환경 | 1cm에 해당하는 픽셀 (대략) |
|---|---|
| 일반 24인치 FHD (96 DPI) | 약 38px |
| 노트북 14인치 FHD | 약 38px |
| 4K 27인치 (157 DPI) | 약 62px |
| 사용자가 브라우저 줌 125% | × 1.25배 |

➡️ **결론**: 사용자가 말한 "cm"은 **본인 모니터 기준의 체감 단위**이며, **정확한 cm 변환은 불가능**합니다.
대신 **현재 폭 대비 비율**과 **사용자의 의도(등급/구분 확대 + 제품명 축소)** 를 충실히 반영합니다.

### 4.2 사용자 요청 해석 (체감 기준)

| 항목 | 사용자 표현 | 추정 의미 | 권장 적용 |
|---|---|---|---|
| 등급 현재 폭 | ~3cm | 90px | (현재) |
| 등급 목표 폭 | ~6cm (7cm는 너무 큼) | **180px** (현재의 2배) | `width:180px` |
| 구분 현재 폭 | ~3cm | 88px | (현재) |
| 구분 목표 폭 | ~4cm | **120px** (현재의 1.36배) | `width:120px` |
| 제품명 변화 | "왼쪽으로 4cm 이동" | 제품명 폭 ~150px 축소 | `width:auto`로 자동 흡수 |

현재 등급 컬럼은 `f-text-xs` (≈11px)로 작은 편입니다.
컬럼 폭을 늘리면 **글자 크기는 그대로 두는 게 안전**합니다 (행 높이 영향 없음).

### 6.3 데이터 검증 (실제 들어 있는 텍스트 길이)

```sql
-- 등급 컬럼에 들어 있는 가장 긴 텍스트
SELECT category, assurance_level, LENGTH(assurance_level) as len
FROM progress_items
ORDER BY len DESC LIMIT 5;
```

**예상 결과** (이미지에서 확인된 것):
- "정보보호시스템" (7자) ← 잘림 발생
- "네트워크장비" (6자) ← 잘림 발생
- "소스코드 보안약점 분석" (11자) ← 215번 행에서 확인 (잘림 심함)
- "EAL1/EAL2/EAL3" (단축형)
- "1" (숫자)

→ **220번 행 "소스코드 보안약점 분석" (11자)** 은 180px로도 부족할 수 있음.
이 경우엔 별도로 짧은 별칭(예: "소스코드 분석")을 데이터 측에서 정리하는 게 근본책.
