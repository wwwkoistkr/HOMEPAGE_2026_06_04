## 📌 결론 먼저: "진짜 원인"은 **반올림 손실**입니다

사용자 증상: **"관리자 모드 AI 시뮬레이터에서 EAL2/EAL3/EAL4의 CCRA 평가일수와 KOIST 평가 프로세스 기간을 입력·저장해도 홈페이지 상단 AI 시뮬레이터에 전혀 반영되지 않는다."**

원인은 **두 단계의 극단적인 숫자 압축(주→월 변환 + 0.40/0.60 분배)을 거치면서, 정수 반올림 때문에 입력값의 변화가 최종 화면 값에 대부분 흡수되어 사라지기 때문**입니다. DB에는 정상적으로 저장되고 있으며, 서버는 해당 값을 읽어 HTML로 렌더링하고 있습니다. 그러나 렌더링 과정의 **정수 반올림 계단** 탓에 **주(week) 단위 4~5주 폭 정도의 수정**은 시뮬레이터 결과에 거의 영향을 주지 못합니다.

가 **사실상 전혀 바뀌지 않음**.

## 2. 실사 증거 (Production HTML ↔ DB 대조)

### 2.1 현재 원격 D1 DB 값 (실사 채취)

```
slug      | trad_min | trad_max | koist_min | koist_max | updated_at
----------+----------+----------+-----------+-----------+--------------------
cc-eal2   |    20    |    36    |     4     |    16     | 2026-04-21 01:24:54
cc-eal3   |    28    |    48    |     4     |    20     | 2026-04-21 01:25:11
cc-eal4   |    36    |    64    |     4     |    20     | 2026-04-21 01:25:23
```

✅ `updated_at` 타임스탬프가 방금 전(04-21 01:24~01:25)입니다.  
→ 관리자가 **실제로 값을 수정·저장**했고, DB에는 **정상적으로 기록**되었습니다.  
→ 따라서 "저장이 안 되는 문제"나 "다른 메뉴를 잘못 편집한 문제"는 **아닙니다.**

### 2.2 현재 프로덕션 HTML에 박혀 있는 `ealData` (실사 채취)

프로덕션 사이트(`https://koist-website.pages.dev/`)의 HTML 소스를 직접 받아 추출한 값:

```js
var ealData = {
  overall: { general:{prep:6,eval:5}, koist:{prepMin:0,prepMax:2,evalMin:1,evalMax:3} },
  EAL2:    { general:{prep:4,eval:4}, koist:{prepMin:0,prepMax:2,evalMin:1,evalMax:2} },
  EAL3:    { general:{prep:6,eval:5}, koist:{prepMin:0,prepMax:2,evalMin:1,evalMax:3} },
  EAL4:    { general:{prep:8,eval:7}, koist:{prepMin:0,prepMax:2,evalMin:1,evalMax:3} }
};
```

✅ **DB의 최신 값이 실제로 HTML에 반영되고 있습니다.** 캐시 문제가 아닙니다.

### 2.3 숫자 범위

- `koist` 쪽 숫자가 **0~3 사이의 한 자리 정수**로 수축되어 있습니다.
- 관리자가 입력한 원본은 "주(week) 단위"지만, 화면에 쓰이는 값은 "개월(month) 단위"로 **두 번 압축**됩니다.

이 압축 과정에서 정수 반올림이 발생해 **입력 감도가 극단적으로 낮아지는 것이 이 문제의 본질**입니다.

## 3. 렌더링 파이프라인 (입력 → 화면)

`src/templates/home.tsx` 53–106줄을 기반으로 도식화합니다.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ [관리자 입력 - 주(weeks) 단위]                                              │
│   CCRA 평가일수:  traditional_min_weeks, traditional_max_weeks              │
│   KOIST 기간:     koist_min_weeks,       koist_max_weeks                    │
└──────────────────────────────┬──────────────────────────────────────────────┘
                               ↓ DB 저장 (정상)
┌─────────────────────────────────────────────────────────────────────────────┐
│ STEP 1 — 주 → 월 변환  (W2M = 4.33주/월)                                     │
│   tradMinM = round(trad_min / 4.33)    ← ⚠️ 실제로 아래 계산에 미사용!      │
│   tradMaxM = round(trad_max / 4.33)                                          │
│   koistMinM = round(koist_min / 4.33)                                        │
│   koistMaxM = round(koist_max / 4.33)                                        │
└──────────────────────────────┬──────────────────────────────────────────────┘
                               ↓ 1차 반올림 후
┌─────────────────────────────────────────────────────────────────────────────┐
│ STEP 2 — 0.40 / 0.60 (또는 0.55 / 0.45) 분배                                 │
│   gPrep    = round(tradMaxM × 0.55)                                          │
│   gEval    = round(tradMaxM × 0.45)                                          │
│   kPrepMin = round(koistMinM × 0.40)    ← ⚠️ 2차 반올림                      │
│   kEvalMin = round(koistMinM × 0.60)                                         │
│   kPrepMax = round(koistMaxM × 0.40)                                         │
│   kEvalMax = round(koistMaxM × 0.60)                                         │
└──────────────────────────────┬──────────────────────────────────────────────┘
                               ↓ 2차 반올림 후
┌─────────────────────────────────────────────────────────────────────────────┐
│ STEP 3 — 클라이언트 JS에서 슬라이더(1~100)로 보간                           │
│   t = 1 − (prepVal − 1) / 99                                                 │
│   kPrep = round(lerp(prepMin, prepMax, t))    ← ⚠️ 3차 반올림                │
│   kEval = round(lerp(evalMin, evalMax, t))                                   │
│   reduction = round((1 − (kPrep+kEval) / (gPrep+gEval)) × 100)               │
└──────────────────────────────┬──────────────────────────────────────────────┘
                               ↓
                         [화면 표시]
```

### 🔑 핵심 포인트

1. **총 3번의 `Math.round()` 반올림**이 순차적으로 적용됨 (1차: 주→월, 2차: 0.4/0.6 분배, 3차: lerp 결과)
2. `koist_min_weeks`는 4 → 1개월 → 0.4개월 → **0**(반올림)이 됨 (즉, EAL2/EAL3/EAL4 모두 `prepMin = 0`)
3. **마지막에 `reduction = round(× 100)`에 다시 한 번 반올림**이 들어가 "5% 단위"로 값이 튐

## 4. 반올림 손실 정량 분석 (감도 맵)

### 4.1 `koist_min_weeks` 감도 (사전준비 100%일 때)

| 입력값(주) | 월(round) | prepMin(×0.4, round) | evalMin(×0.6, round) |
|:---:|:---:|:---:|:---:|
| **1~2** | 0 | **0** | **0** |
| **3** | 1 | 0 | **1 ← 변화** |
| **4~6** | 1 | 0 | 1 |
| **7** | 2 | **1 ← 변화** | 1 |
| **8~10** | 2 | 1 | 1 |
| **11** | 3 | 1 | **2 ← 변화** |
| 12~15 | 3 | 1 | 2 |
| **16** | 4 | **2 ← 변화** | 2 |
| 17~19 | 4 | 2 | 2 |
| **20** | 5 | 2 | **3 ← 변화** |
| 21~23 | 5 | 2 | 3 |
| **24** | 6 | 2 | **4 ← 변화** |

**→ `koist_min_weeks`는 오직 1, 3, 7, 11, 16, 20, 24, 29, 33, ... (즉 "~4주마다 한 번") 경계에서만 결과가 바뀜.**  
나머지는 모두 **입력 변경이 화면에 반영되지 않음**.

### 4.2 `koist_max_weeks` 감도 (사전준비 1%일 때)

| 입력값(주) | 월 | prepMax | evalMax |
|:---:|:---:|:---:|:---:|
| 1~2 | 0 | **0** | **0** |
| 3~6 | 1 | 0 | 1 |
| 7~10 | 2 | 1 | 1 |
| 11~15 | 3 | 1 | 2 |
| 16~19 | 4 | 2 | 2 |
| 20~23 | 5 | 2 | 3 |
| 24~28 | 6 | 2 | 4 |
| 29~32 | 7 | 3 | 4 |
| 33~36 | 8 | 3 | 5 |
| 37~40 | 9 | 4 | 5 |

**→ `koist_max_weeks`도 약 "4주 단위 계단"에서만 결과가 튐.**

### 4.3 EAL2 기준 실제 시뮬레이션 (슬라이더=50 위치, reduction 비교)

```
 koist_min  koist_max  →  prepMin prepMax evalMin evalMax  |  reduction
─────────── ───────── ──── ─────── ─────── ─────── ───────  ─  ─────────
    3주      12주          0       1       1       2        |    62%
    3주      14주          0       1       1       2        |    62%    ← 동일
    3주      16주          0       2       1       2        |    62%    ← 동일
    3주      18주          0       2       1       2        |    62%    ← 동일
    3주      20주          0       2       1       3        |    62%    ← 동일
    4주      12주          0       1       1       2        |    62%    ← 동일
    4주      16주          0       2       1       2        |    62%    ← 현재값
    4주      20주          0       2       1       3        |    62%    ← 동일
    5주      20주          0       2       1       3        |    62%    ← 동일
    6주      20주          0       2       1       3        |    62%    ← 동일
```

**→ `koist_min_weeks`를 3~6주 범위, `koist_max_weeks`를 12~20주 범위에서 어떻게 조합해도 reduction은 62% 고정!**

즉, 관리자가 EAL2의 KOIST 기간을 "4주~16주" → "5주~18주" → "3주~14주" 등으로 여러 번 바꿔도 **슬라이더 정 가운데에서는 결과가 **완전히 동일**하게 62%로 표시**됩니다. 사용자가 겪은 증상과 **정확히 일치**합니다.

## 5. 추가로 발견된 구조적 문제

### 5.1 `traditional_min_weeks`는 **완전히 미사용**

`simTypeToEal()` 함수(home.tsx 56–70줄) 본문을 다시 보면:

```ts
const tradMinM = Math.round(st.traditional_min_weeks / W2M);  // ← 계산만 됨
const tradMaxM = Math.round(st.traditional_max_weeks / W2M);
...
const gPrep = Math.round(tradMaxM * 0.55);  // ← tradMaxM만 사용
const gEval = Math.round(tradMaxM * 0.45);  // ← tradMaxM만 사용
```

- 변수 `tradMinM`이 계산되지만 **`return` 객체에 포함되지 않으며 어디에서도 사용되지 않습니다.**
- 즉, 관리자가 **"CCRA평가일수 - 최소(사전준비 100%)"** 필드를 아무리 수정해도 **결과값에 0%의 영향**을 줍니다. 이것은 말 그대로 "유령 필드"입니다.

### 5.2 일반(CCRA) 기간은 슬라이더와 무관

일반(`general`) 기간은 `gPrep`, `gEval` 두 개의 상수값으로만 쓰입니다. 슬라이더를 이동해도 일반(CCRA) 막대는 **항상 동일한 고정 기간**으로 표시됩니다. 이는 설계상 의도일 수 있으나, 사용자는 "CCRA도 슬라이더와 함께 변하는 것"이라 기대할 가능성이 있습니다.

### 5.3 분배 비율(0.55/0.45, 0.40/0.60)의 하드코딩

- 일반: 준비 55%, 평가 45%
- KOIST: 준비 40%, 평가 60%

이 비율은 어디에도 관리자 인터페이스가 없으며 코드에 하드코딩되어 있습니다. 실제 평가 프로세스가 이와 다른 비율을 가진다면, 입력한 총 기간이 옳더라도 **세부 막대 비율이 실제와 다르게 표시**됩니다.

### 5.4 EAL 시뮬레이터는 **오직 `cc-eal2` / `cc-eal3` / `cc-eal4` 3개 slug만 참조**

코드 73–75줄:
```ts
const eal2Type = simTypes.find(t => t.slug === 'cc-eal2');
const eal3Type = simTypes.find(t => t.slug === 'cc-eal3');
const eal4Type = simTypes.find(t => t.slug === 'cc-eal4');
```

- `overall`은 위 3개의 평균으로 계산됩니다.
- 관리자 요구사항대로 **KCMVP / 보안기능확인서 / 성능평가는 무관**합니다. (올바른 설계)

### 5.5 헤더 Badge(%) 초기값 고정

- 페이지 로드 시 `#headerReductionPct`와 `#ealReductionText`는 `unified_reduction_default`(DB 기본 35%)를 그대로 표시합니다(home.tsx 591, 675줄).
- **슬라이더를 1회라도 움직여야** `updateChart()`가 호출되어 실제 계산값으로 갱신됩니다.
- 즉, 관리자가 값을 바꾸고 홈페이지에 접속해도 **로드 직후 화면에는 "35%"가 고정 표시**되므로, 사용자는 "값이 하나도 안 바뀌었다"고 오해하기 쉬운 구조입니다.

### 5.6 Cloudflare Edge Cache

- 저장 후 약 수 초~수 분간 이전 HTML이 캐시될 수 있습니다.
- URL 뒤에 `?v=타임스탬프`를 붙이면 바로 우회됩니다.
- 이번 실사에서 캐시 우회 후 조회한 결과 **DB 최신값이 정상 렌더**되고 있었으므로, 본 증상의 주원인은 캐시가 아닙니다.

## 7. 진단을 위한 검증 시나리오 (코드 수정 없이 재현 가능)

### 시나리오 A — 반올림 손실 입증 (현 버전에서 "안 바뀌는" 경우)
1. `/admin/sim-cert-types` → CC평가 (EAL2) [수정]
2. KOIST 기간: 최소 **4** → **5**, 최대 **16** → **18**로 변경, 저장
3. 홈페이지 `https://koist-website.pages.dev/?v=A1` 접속
4. EAL2 탭 선택 후 슬라이더를 50 위치에 놓음
5. **→ reduction 62%가 그대로. 막대 길이, 준비/평가 개월수 전부 동일.**  
   (본 보고서 §4.3 참조)

### 시나리오 B — 반올림 경계 돌파 (현 버전에서도 "변하는" 경우)
1. `/admin/sim-cert-types` → CC평가 (EAL2) [수정]
2. KOIST 기간: 최소 **4** → **11**(※ 4주 계단 2개 이상 점프), 최대 **16** → **32**, 저장
3. 홈페이지 `?v=B1` 접속
4. **→ reduction 숫자, 막대 길이, 개월수가 비로소 바뀜**.  
   이는 주→월 반올림 계단을 2칸 이상 건너뛰어서 계산값에 변화가 누적되었기 때문.

### 시나리오 C — `traditional_min_weeks`가 유령 필드임을 확인
1. `/admin/sim-cert-types` → CC평가 (EAL2) [수정]
2. CCRA평가일수: **최소 20 → 1** (다른 값은 고정), 저장
3. 홈페이지 `?v=C1` 접속 → **완전히 동일한 화면** 표시  
   (코드에서 `tradMinM`이 어디에도 사용되지 않으므로, 아무 변화 없음)

### 시나리오 D — 헤더 Badge 초기값 고정 확인
1. `/admin/home-content` → "기본 단축률" 35 → 80, 저장
2. 홈페이지 `?v=D1` 접속
3. **슬라이더를 움직이기 전**: 우측 상단 Badge **"80%"** 표시 (실제 계산과 무관)
4. 슬라이더를 1회 이동 → Badge가 **실제 계산값(예: 62%)** 으로 갱신됨

## 8. 권장 해결 방향 (요청 시 별도 작업)

### 8.1 🔴 단기(필수) — 반올림 손실 제거
- 내부 계산을 **주(week) 단위** 그대로 유지하고, **최종 표시 단계에서만** 개월로 변환
- 혹은 **소수점 한 자리까지 유지** (`Math.round(x)` → `Math.round(x * 10) / 10`)
- 구현 난이도: 낮음 (home.tsx의 `simTypeToEal()` 함수 약 10줄 수정)

### 8.2 🔴 단기(필수) — `traditional_min_weeks` 활용 또는 필드 숨김
- 관리자 UI에서 사용 중인 필드가 실제로는 무시되고 있으니, 계산식에 반영하거나 UI에서 제거 (관리자 혼선 방지)
- 구현 난이도: 낮음

### 8.3 🟠 단기 — 헤더 Badge(%) 초기값을 서버사이드에서 계산
- 페이지 렌더 시점에 슬라이더=50 기준의 `reduction` 값을 계산해 초기 DOM에 주입
- 구현 난이도: 낮음 (home.tsx 122줄 정도)

### 8.4 🟠 중기 — 분배 비율(0.55/0.45, 0.40/0.60) 관리자화
- `site_settings` 또는 `sim_cert_types`에 컬럼을 추가해 비율을 편집 가능하게 함
- 구현 난이도: 중간 (DB 마이그레이션 + UI 추가)

### 8.5 🟡 중기 — 관리자 저장 시 해당 페이지 Cloudflare Cache purge
- Cloudflare API로 저장 직후 `/` 경로 캐시 퍼지
- 구현 난이도: 중간

### 8.6 🟡 중기 — 관리자 UI에 "실시간 미리보기" 추가
- 저장 전에 입력값이 슬라이더 결과에 어떤 영향을 주는지 관리자 화면 옆에 실시간 계산해 보여줌
- 구현 난이도: 중간
