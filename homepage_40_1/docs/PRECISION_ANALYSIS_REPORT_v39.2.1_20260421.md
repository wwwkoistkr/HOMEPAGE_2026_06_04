## 4. ⚠️ 단일 근본 원인: 표시 레이어의 정수 반올림 왜곡

### 4-1. 문제 코드 위치 (home.tsx 1927~1946)

```javascript
// CCRA 표시 — Math.round() 직접 사용
gTotal.textContent = '약 ' + Math.round(d.general.total) + '개월';   // 라인 1927
gPrep.textContent  = '준비 ' + Math.round(d.general.prep)  + '개월'; // 라인 1929
gEval.textContent  = '평가 ' + Math.round(d.general.eval)  + '개월'; // 라인 1931

// KOIST 표시 — fmtM() (결국 Math.round 동일)
kTotal.textContent = '약 ' + fmtM(d.koist.total) + '개월';  // 라인 1942
kPrep.textContent  = '준비 ' + fmtM(d.koist.prep)  + '개월'; // 라인 1944
kEval.textContent  = '평가 ' + fmtM(d.koist.eval)  + '개월'; // 라인 1946

function fmtM(v) { return Math.round(v).toString(); }  // 라인 1901
```

**핵심 문제**: **총합(Total)**과 **분할값(Prep/Eval)**을 각각 독립적으로 반올림한 뒤, 그대로 화면에 표시. 하지만 사용자는 시각적으로 "준비 + 평가 = 총합"으로 읽기 때문에 괴리가 발생.

### 4-2. 수학적 원인 — Rounding Sum Inequality

일반적으로 실수 a, b에 대해:
- `round(a) + round(b) ≠ round(a + b)` (각각 0.5 이내 오차 → 합해서 최대 1 오차 가능)

예시 (EAL2, Prep=1%):
```
실수:   gPrep = 4.60,   gEval = 3.70,   합 = 8.30
반올림: 준비 5,         평가 4,         총 8
시각:   5 + 4 = 9  ← 사용자가 보는 값
표시:   약 8개월     ← 시스템이 보여주는 값
```
→ **1개월 차이**로 사용자는 "계산이 틀렸다" 또는 "바가 이상하다"고 오인.

## 7. 왜 "바는 안 움직이는 것처럼 보이는가"의 체감 분석

사용자가 관찰하는 현상을 단계별로 해석:

1. **숫자는 바뀐다**(실제로 바뀜)
2. **그런데 준비+평가가 총합과 안 맞는다** → 계산이 이상하다고 의심
3. **시선이 숫자 → 바 → 숫자 왔다 갔다** 하는 중에 바의 width 변화(예: 89% → 78%, 11%p 차이)가 **시각적으로 크지 않아서** 느린 애니메이션(0.7초)과 겹쳐 "안 움직이는 것처럼" 느껴짐
4. 특히 **슬라이더 중간 구간(25~75%)** 에서는 바 너비 차이가 11%p(≈78px) 수준이라 체감이 약함

**결론**: 바는 실제로 움직이고 있으나, **숫자 표시의 정수 왜곡이 "바까지 이상하다"는 착시**를 유발.

| **C. Total을 Prep+Eval의 반올림 합으로 계산** | `total = round(prep) + round(eval)` | 덧셈 맞음 | 총합이 실제값과 살짝 다를 수 있음 |
| **D. 올림(ceil) 통일** | 모든 값에 `Math.ceil` 적용 | 안전하게 긴 쪽 표시 | 실제보다 항상 긺, 단축률과 모순 생길 수 있음 |
| **E. 표시는 정수, 툴팁에 소수 노출** | 디자인 유지 + 상세값 제공 | 타협안 | 구현 필요 |

> **Option B (소수점 1자리 표시)** 또는 **Option C (Total=P+E 강제 일치)**
